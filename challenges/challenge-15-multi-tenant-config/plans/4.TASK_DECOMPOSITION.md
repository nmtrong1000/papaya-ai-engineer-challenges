## Notes

- **MTC-01** creates the npm workspace root at `app/` and two of its three members: the `app/shared` skeleton (package.json + empty `src/index.ts`) and `app/frontend` (Next.js App Router + Tailwind). The `@mtc/shared` path alias is configured in the frontend's `tsconfig.json`. The backend is not part of this task.
- **MTC-02** creates the third workspace member: `app/backend` (Express + TypeScript + `ts-node-dev`). It connects to `app/shared` via the same `@mtc/shared` path alias so both consumers resolve shared types identically. The workspace root already exists from MTC-01; this task only adds the backend package and runs `npm install` to re-link workspaces.
- **MTC-03** writes all 8 Prisma models (`Tenant`, `Branding`, `ClaimType`, `TenantClaimType`, `TenantNotification`, `ApprovalTier`, `CustomField`, `TenantVersion`) and runs the initial migration against Neon.
- **MTC-04** writes `prisma/seed.ts` for SafeGuard Insurance, HealthFirst, and GovHealth with meaningfully different configs across all 6 sections.
- **MTC-05** writes the Zod schemas in `app/shared`: `TenantConfigSchema`, `VersionConfigSchema`, `ClaimDataSchema`, `ProcessClaimResultSchema`. These are the single source of truth consumed by both backend validation and frontend form validation.
- **MTC-06** writes the full OpenAPI 3.0 specification as `@swagger` JSDoc annotations in dedicated `app/backend/src/docs/*.docs.ts` files (no route logic, spec only). swagger-jsdoc picks these up via glob. The `/api-docs` Swagger UI endpoint is wired here and serves the complete spec before any route is implemented. This contract is the source of truth for both frontend (what to call) and backend (what to build) — MTC-07 through MTC-09 implement to match it, and MTC-10 through MTC-16 consume it.
- **MTC-07** implements all tenant CRUD routes using a three-layer architecture: `TenantRepository` (Prisma calls per table), `TenantService` (8-step transactional save flow, config cache, snapshot building), `TenantController` (HTTP parsing/response), `routes/tenants.ts` (path definitions only). Every PUT and POST runs the full save transaction: upsert branding, update threshold, delete+reinsert tiers / claim types / notifications / custom fields, insert `TenantVersion` snapshot, update `currentVersionId`.
- **MTC-08** adds version history and rollback using the same three-layer structure: `VersionRepository` (Prisma queries, omits `config` JSON from list), `VersionService` (list + rollback logic via `migrateConfig()` + `TenantService.saveTenantConfig`), `VersionController`, `routes/versions.ts`. Rollback creates a new version and never mutates history.
- **MTC-09** implements `ClaimService.processClaim(config, claimData)` as a **pure function** (no Prisma, no HTTP). `ClaimController` loads the tenant config from `TenantService.getTenantById()` (cache-first) and delegates processing to `ClaimService`. No repository is needed — no DB write occurs during claim processing.
- **MTC-10** covers `TenantService` and `ClaimService` with Vitest unit tests. Prisma and the config cache are mocked; no HTTP involved. Target ≥ 60% line coverage across both service files.
The frontend uses a **layer-based architecture**: `app/` pages are thin (wire hooks to components only); `components/` are pure UI (props in, callbacks out, no fetch); `hooks/` own all data fetching and state; `lib/` contains the typed `apiFetch` wrapper.

- **MTC-11** creates `hooks/useTenantList` (GET + delete), `components/TenantList` (pure UI table), the sidebar layout, and the `/tenants` list page (thin: uses hook, renders component).
- **MTC-12** creates `hooks/useTenant` (GET single config for edit pre-fill), builds the pure UI `TenantForm` with `BrandingSection` and `ClaimTypesSection`. Edit page is thin: uses `useTenant`, passes config as `defaultValues` to `TenantForm`.
- **MTC-13** adds three more pure UI sections to `TenantForm`: `ApprovalRulesSection`, `NotificationsSection`, `CustomFieldsSection` — all using `useFieldArray`, no fetch.
- **MTC-14** creates `hooks/useTenantMutations` (`useCreateTenant`, `useUpdateTenant`) that own POST/PUT calls and redirect. Pages pass the hook's `submit` as `onSubmit` to `TenantForm` — the form component remains unaware of which API is called.
- **MTC-15** creates `hooks/useProcessClaim` (POST process-claim), `components/ClaimResult` (pure UI result display), and the thin `/preview` page that wires them together.
- **MTC-16** creates `hooks/useDiff` (parallel fetch of two tenant configs), `components/DiffTable` + `DiffRow` (pure UI), and the thin `/diff` page.
- **MTC-17** creates `hooks/useVersionHistory` (GET versions + POST rollback), `components/VersionHistory` (pure UI table with rollback button), and adds a tab UI to the edit page — rollback success re-fetches config and resets the form.
- **MTC-18** covers all API routes (CRUD, versions, rollback, process-claim) with Vitest + Supertest integration tests against the live Neon test database. No mocking of Prisma or HTTP. Target ≥ 60% line coverage across controllers and routes.
- **MTC-19** and **MTC-20** are the two-step deploy. Backend first (Fly.io + Dockerfile + env secrets), then frontend (Vercel + `NEXT_PUBLIC_API_URL` env var). The Dockerfile for Fly.io copies from the `app/` root so it can access `app/shared` alongside `app/backend`. Smoke test in MTC-20 verifies all pages work against the live backend with the 3 seeded tenants.
