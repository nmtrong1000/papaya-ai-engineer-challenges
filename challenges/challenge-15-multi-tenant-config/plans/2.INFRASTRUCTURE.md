# 2. Infrastructure

## 2.1 Frontend

### Framework
- [ ] **React + Vite** — SPA with a separate API server. Clean separation but adds deployment complexity (two services). **Why consider:** Familiar, lightweight, great DX. **Trade-offs:** No built-in API routes; need a separate backend service; more configuration overhead.
- [x] **Next.js (App Router)** — Full-stack React framework with built-in API route handlers, SSR, and file-based routing. **Why consider:** Single deployment unit covers both the admin UI and the runtime API; native Vercel support; App Router makes co-locating API and UI trivial. **Trade-offs:** More opinionated than Vite; App Router has a steeper learning curve than Pages Router.
- [ ] **Remix** — Full-stack framework with progressive enhancement focus. **Why consider:** Excellent form handling, nested routes, loader/action pattern is clean for CRUD. **Trade-offs:** Smaller ecosystem than Next.js; less tooling support; Vercel deployment works but Next.js is a better fit there.

### Styling
- [ ] **Tailwind CSS + shadcn/ui** — Utility-first CSS with a copy-paste component library (Button, Dialog, Table, Form, Tabs, Badge) built on Radix UI primitives. **Why consider:** Admin UIs have complex forms, dialogs, and data tables — shadcn provides all of these production-ready with full accessibility. No runtime overhead. **Trade-offs:** Verbose class names; shadcn components need to be added one by one.
- [x] **Tailwind CSS only** — Utility classes without a component library. **Why consider:** More control, no third-party component constraints. **Trade-offs:** Need to hand-roll every form, dialog, and table for a complex admin UI — slow for a timed challenge.
- [ ] **CSS Modules** — Scoped CSS per component. **Why consider:** Zero dependencies, no class name bloat. **Trade-offs:** No pre-built components; significant styling time for a complex admin UI.

### Form Handling & Validation
- [x] **React Hook Form + Zod** — RHF handles complex nested form state (arrays of tiers, arrays of notification events); Zod provides schema-based validation shared between frontend and backend. **Why consider:** The tenant config form has deeply nested arrays (approval tiers, SLA per claim type, custom fields) — RHF's `useFieldArray` is purpose-built for this. Zod schemas double as the runtime type guards. **Trade-offs:** Two libraries to learn; overkill for simple forms.
- [ ] **Formik + Yup** — Older but established combo. **Why consider:** Widely documented. **Trade-offs:** Heavier than RHF; re-renders entire form on every keystroke without optimization; Yup is less ergonomic than Zod for TypeScript.
- [ ] **Uncontrolled HTML forms (native)** — No library. **Why consider:** Zero dependencies. **Trade-offs:** Managing dynamic arrays of approval tiers and custom fields with native forms is error-prone and tedious.

## 2.2 Backend

- [ ] **Next.js Route Handlers** — API endpoints defined as `app/api/[...]/route.ts` files. **Why consider:** Same repo, same deployment, no CORS setup, shared types with the frontend. The `processClaim()` function lives here as a plain TypeScript module. **Trade-offs:** Not suitable for CPU-intensive workloads; tied to Vercel's serverless execution model.
- [x] **Express.js (separate server)** — Traditional Node.js server deployed independently from the frontend. **Why consider:** Maximum flexibility, widely understood, no framework constraints. The `processClaim()` function becomes a plain REST endpoint callable from any frontend. **Trade-offs:** Two services to deploy and maintain; CORS must be configured; environment variables must be managed in two places; no shared types without a monorepo or shared package.
- [ ] **Hono** — Lightweight edge-compatible framework. **Why consider:** Extremely fast, runs on Cloudflare Workers / Vercel Edge. **Trade-offs:** Less ecosystem maturity; unnecessary complexity when Next.js Route Handlers already provide what's needed.

---

## 2.3 Database

- [x] **PostgreSQL via Neon (+ Prisma ORM)** — Serverless PostgreSQL with a generous free tier. Prisma provides typed schema, migrations, and query builder. **Why consider:** Relational model is ideal for `Tenant → ConfigVersion` (one-to-many with ordered history). Neon's free tier has no sleep delays. Prisma's `@default(now())` and `@@map` make version history trivial. **Trade-offs:** Requires schema migration step; slightly more setup than a document DB.
- [ ] **MongoDB via Atlas (+ Mongoose or Prisma)** — Document database. **Why consider:** Tenant config is naturally a JSON document; no schema migration needed for config shape changes. Free tier available. **Trade-offs:** Relational queries (e.g., "all versions of tenant X ordered by time") are less clean; Atlas free tier has a 512MB limit and potential cold-start pauses.
- [ ] **SQLite via Turso** — Distributed SQLite with an HTTP API. **Why consider:** Zero operational overhead, free tier, excellent Prisma support. **Trade-offs:** Less familiar; Turso's edge replication is overkill for this challenge; local SQLite file won't work in serverless.

## 2.4 DevOps / Deployment

### Option A — Single service (Next.js full-stack)

- [ ] **Vercel** — Native Next.js hosting. Zero-config deploy from GitHub. Automatic preview deployments per PR. **Why consider:** Free tier covers this challenge; native support for Next.js serverless functions; environment variables UI is straightforward. **Trade-offs:** Vendor lock-in on serverless function runtime; 10s timeout on free tier (sufficient for this use case).

### Option B — Two services (Express backend + separate frontend)

If Express is chosen as the backend, the frontend and backend must be deployed independently. The frontend URL must be set as a CORS origin on the Express server, and the backend URL must be set as an environment variable on the frontend.

| Service | Hosts | Cold Start | Free Tier |
|---------|-------|-----------|-----------|
| **Railway** | Express backend + PostgreSQL (same project) | None | $5/month credit |
| **Render** | Express backend | ~30s after 15 min idle | Yes (with sleep) |
| **Fly.io** | Express backend (containerized) | None | Yes (needs Dockerfile) |

- [ ] **Vercel (frontend) + Railway (backend + DB)** — Most practical two-service combo. Railway runs the Express app and Postgres in one project; Vercel hosts the frontend. **Why consider:** Railway has no cold-start sleep; $5 credit covers light usage; built-in Postgres saves a separate DB service. **Trade-offs:** Two platforms to manage; Railway free credit can run out under heavy use.
- [ ] **Vercel (frontend) + Render (backend)** — Common pairing. **Why consider:** Both have generous free tiers. **Trade-offs:** Render's free web services sleep after 15 min of inactivity — a ~30s cold start on first request, which is noticeable during a live demo or review.
- [x] **Vercel (frontend) + Fly.io (backend)** — Container-based backend with no cold-start sleep. **Why consider:** Reliable uptime on free tier; Docker-based so fully portable. **Trade-offs:** Requires a `Dockerfile`; slightly more setup than Railway; less straightforward for a timed challenge.

## 2.5 External Services / APIs

No external APIs are required. All processing is local:
- Notification channels (email, SMS, webhook) are **computed and returned**, not dispatched — no SendGrid, Twilio, or webhook infrastructure needed.
- No authentication provider required — the admin UI has no login.

## 2.6 API Documentation

Applies only if **Express** is chosen as the backend. Next.js Route Handlers do not need a separate API doc layer — the TypeScript types serve as the contract within the same codebase.

- [ ] **`@asteasolutions/zod-to-openapi` + `swagger-ui-express`** — Generates an OpenAPI 3.x spec automatically from the existing Zod schemas (the same schemas used for form validation and `processClaim()` input/output). A Swagger UI is served at `/api-docs`. **Why consider:** Single source of truth — the spec is derived from the schemas, so it cannot drift from the runtime behaviour. Zero duplication: define the shape once, get both validation and documentation. **Trade-offs:** Requires registering each schema with the OpenAPI registry (a few extra lines per schema); less familiar than JSDoc comments for developers new to zod-to-openapi.
- [x] **`swagger-jsdoc` + `swagger-ui-express`** — OpenAPI spec written as JSDoc comments directly above each Express route handler; `swagger-jsdoc` parses them at startup and serves a Swagger UI. **Why consider:** Widely used and documented; spec lives next to the route it describes. **Trade-offs:** The JSDoc comment and the actual handler are separate artefacts that can silently drift out of sync; more verbose than schema-derived generation.
- [ ] **`tsoa`** — Generates both the OpenAPI spec and the Express route registration from TypeScript decorators on controller classes. The spec is always accurate because it is derived from the compiled code at build time. **Why consider:** Tightest accuracy guarantee; routes and spec are one artefact. **Trade-offs:** Requires restructuring routes into controller classes; adds a build step (`tsoa spec-and-routes`); most setup overhead of the three options.

## Conclusion

The selected stack is **Next.js (App Router) + Tailwind CSS + React Hook Form + Zod** on the frontend, **Express.js + Prisma + Neon PostgreSQL** on the backend, deployed as two services — **Vercel** for the frontend and **Fly.io** for the backend. API contracts are documented via **swagger-jsdoc + swagger-ui-express**.

The frontend and backend live in a single monorepo under `app/frontend` and `app/backend`, with a `app/shared` directory for Zod schemas and TypeScript types shared between both sides — eliminating duplication and keeping the API contract in sync with the runtime.

Next.js (App Router) was chosen for the frontend because it provides SSR and file-based routing for the admin UI while coexisting cleanly with the separate Express backend. Tailwind CSS without a component library gives full control over the UI without third-party constraints, accepting the trade-off of hand-rolling form and table components. React Hook Form with Zod handles the deeply nested config forms (arrays of approval tiers, per-claim-type SLA entries, dynamic custom fields) with full type safety. Express provides a familiar, flexible REST layer for the `processClaim()` API with no framework constraints. Prisma + Neon gives a reliable relational model for the `Tenant → ConfigVersion` history chain with no infrastructure management. Fly.io hosts the Express backend with no cold-start sleep, making the live demo reliable. swagger-jsdoc documents the API inline with route handlers, keeping the spec close to the code.
