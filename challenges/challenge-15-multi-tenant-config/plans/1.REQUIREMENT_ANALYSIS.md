# 1. Requirement Analysis

## 1.1 Problem Statement

Insurance platforms that serve multiple insurers face a fundamental operational challenge: each insurer has distinct branding, claim workflows, approval hierarchies, notification preferences, and SLA targets. Hardcoding these differences requires engineering involvement for every new client onboarding, which is slow and error-prone. This challenge is to build a multi-tenant configuration platform where an operations team member can create and manage tenant configurations entirely through an admin UI — no code changes required — and a runtime engine processes claims according to each tenant's specific rules.

## 1.2 Functional Requirements

### Tenant Configuration — Schema
- [ ] FR-01: Each tenant config stores branding: company name, logo URL, primary color, secondary color
- [ ] FR-02: Each tenant config stores enabled claim types (subset of: OUTPATIENT, INPATIENT, DENTAL, MATERNITY, OPTICAL) with required and optional documents per type
- [ ] FR-03: Each tenant config stores approval rules: auto-approval threshold (amount) and tiered approval tiers (amount range → required approver role)
- [ ] FR-04: Each tenant config stores notification settings: which events trigger notifications (claim_submitted, approved, rejected, payment_sent), which channels per event (email, SMS, webhook), and optional custom email templates per event
- [ ] FR-05: Each tenant config stores SLA settings: target processing time in business days per claim type, and escalation rules (who to notify on breach)
- [ ] FR-06: Each tenant config stores custom fields: additional data fields required at claim submission (name, type, required flag)

### Admin UI
- [ ] FR-07: Create a new tenant configuration via a form in the admin UI
- [ ] FR-08: Read / list all existing tenant configurations
- [ ] FR-09: Update an existing tenant configuration
- [ ] FR-10: Delete a tenant configuration
- [ ] FR-11: Form validation: auto-approval threshold must be ≥ 0, at least one claim type must be enabled, SLA values must be positive integers
- [ ] FR-12: Preview mode — select a tenant and enter a sample claim (type + amount + submission date) → display: required documents, approval tier and approver role, notifications to fire, SLA deadline, custom fields required
- [ ] FR-13: Config diff — select any two tenants → side-by-side view with differing fields highlighted
- [ ] FR-14: Config history — every config save creates a new version; user can view the full version history and rollback to any prior version

### Runtime Processing
- [ ] FR-15: Implement `processClaim(tenantId, claimData)` that returns: required documents, approval routing (tier + approver role), notifications to send (channels), SLA deadline, custom fields required with validation status
- [ ] FR-16: Seed 3 tenants matching the brief's sample tenants (SafeGuard Insurance, HealthFirst, GovHealth) with meaningfully different configurations
- [ ] FR-17: Demonstrate the same claim submitted to all 3 tenants producing different processing outcomes
- [ ] FR-18: A 4th tenant can be onboarded entirely through the admin UI with zero code changes

## 1.3 Non-Functional Requirements

- NFR-01: **Accessibility** — all interactive elements are keyboard-navigable; form inputs have proper labels; color contrast meets WCAG AA
- NFR-02: **Responsiveness** — admin UI is usable on desktop (primary) and tablet; not required to be fully mobile-optimized
- NFR-03: **Persistence** — tenant configurations and version history survive page refresh (database or durable storage)
- NFR-04: **Performance** — config save, list, and load operations complete in under 500ms under normal conditions
- NFR-05: **Modularity** — adding a new configuration dimension (e.g., currency settings) requires changes in one schema/type definition, not scattered across runtime logic
- NFR-06: No unhandled console errors on the golden path

## 1.4 Out of Scope

- Actual email, SMS, or webhook delivery (notifications are computed and displayed, not dispatched)
- Authentication or role-based access control for the admin UI
- Multi-user concurrent editing or conflict resolution
- Production security hardening (API key rotation, rate limiting, input sanitization beyond form validation)
- A real claims processing workflow or approval execution engine — the `processClaim` function is a deterministic routing function, not a workflow orchestrator
- Claim status tracking or a claims inbox UI

## 1.5 Definition of Done

- [ ] CRUD for tenant configurations is fully operational (create, list, edit, delete)
- [ ] All 6 config sections (branding, claim types, approval rules, notifications, SLA, custom fields) are editable through the admin UI
- [ ] Form validation blocks invalid submissions and shows clear error messages
- [ ] Preview mode correctly computes and displays claim processing outcome for any tenant + claim combination
- [ ] Config diff accurately highlights all differences between any two selected tenants
- [ ] Config history records every save; rollback restores the selected version and creates a new history entry
- [ ] 3 sample tenants are seeded with the exact specs from the brief and produce demonstrably different `processClaim` outputs
- [ ] `processClaim(tenantId, claimData)` returns correct required documents, approval tier, notifications, SLA deadline, and custom field validation for all 3 tenants
- [ ] A 4th tenant created via the admin UI can immediately be used to process claims without any code change
- [ ] App is deployed to free hosting with a publicly accessible live URL
- [ ] No console errors on the golden path
