# Phase 2 — Infrastructure

## 2.1 Frontend

### Framework

- [x] **React + Vite** — A component-based SPA using Vite as the build tool. Fast dev server, minimal config, produces a static build that deploys anywhere. **Why consider:** Best balance of component reuse and low setup cost for a single-page challenge. **Trade-offs:** Requires Node.js toolchain; no SSR (not needed here).

- [ ] **Next.js (App Router)** — Full-stack React framework with built-in routing, SSR/SSG, and first-class Vercel deployment. **Why consider:** Zero-config Vercel deploys; App Router enables server components for clean data handling. **Trade-offs:** More framework overhead than the challenge warrants; slower cold starts on free-tier hosting.

- [ ] **Plain HTML + CSS + Vanilla JS** — A single `index.html` with embedded or linked CSS and a JS file to render the table from a JSON constant. **Why consider:** Zero build step, instant deploy to GitHub Pages or Netlify Drop. **Trade-offs:** No component model — styling and logic live in the same files, making the code harder to organize cleanly; no JSX means more verbose DOM manipulation.

### Styling

- [x] **Tailwind CSS** — Utility-first CSS framework applied directly in JSX class names. **Why consider:** Rapid layout iteration; responsive variants (`md:`, `lg:`) and state variants (`hover:`) keep all styling co-located with markup. **Trade-offs:** Large class strings in JSX; requires PostCSS setup.

- [ ] **CSS Modules** — Scoped CSS files imported per component (`*.module.css`). **Why consider:** Zero runtime overhead; class names are locally scoped so no naming collisions. **Trade-offs:** More context-switching between `.tsx` and `.module.css` files; no built-in design token system.

- [ ] **Plain CSS (global stylesheet)** — A single `index.css` with BEM or flat selectors. **Why consider:** No tooling required; works with any build setup. **Trade-offs:** Global scope risks naming collisions; harder to maintain as the component tree grows.

## 2.4 DevOps / Deployment

- [x] **Vercel** — Git-push deploys with automatic preview URLs. Native support for React/Vite and Next.js. Free hobby tier is sufficient. **Why consider:** Fastest path from `git push` to live URL; zero config for Vite or Next.js projects. **Trade-offs:** Vendor lock-in for advanced features (not relevant at this scale).

- [ ] **Netlify** — Drag-and-drop or Git-connected static site hosting. Supports any static build output. **Why consider:** Framework-agnostic; works equally well for plain HTML or Vite builds. **Trade-offs:** Slightly more manual configuration for non-Netlify-native frameworks compared to Vercel.

- [ ] **GitHub Pages** — Free static hosting directly from a GitHub repo branch or `/docs` folder. **Why consider:** No third-party service needed; repo and live URL are in one place. **Trade-offs:** Requires a `base` path config tweak for Vite; no HTTPS custom domain on free tier; deploy pipeline is more manual unless using GitHub Actions.

## 2.5 External Services / APIs

No external APIs are required. All plan data is static and self-contained.

---

## Conclusion

The selected stack is **React + Vite + TypeScript + Tailwind CSS** deployed to **Vercel**. React's component model keeps the comparison table, benefit rows, and badge logic cleanly separated — the most important reason given the evaluation criterion of "code organization and cleanliness." Vite gives a fast dev loop with zero boilerplate. Tailwind CSS keeps responsive layout and highlight states co-located with markup via utility classes, avoiding a separate stylesheet. Vercel is the natural deployment target for Vite projects: a single `git push` to `main` produces a live HTTPS URL with no extra configuration.
